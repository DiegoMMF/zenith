● Zenith es un harness multi-agente para tareas de codificación de larga
  duración. La idea central: un orquestador supervisa el trabajo, lanza
  subagentes (workers/testers), replantea el plan cuando detecta huecos, y
  decide cuándo parar — evitando que el agente se detenga prematuramente.

  Para implementarlo en tu proyecto:

  1. Instalar dependencias
  cd zenith && uv sync
  npm install -g @agentclientprotocol/claude-agent-acp
  2. Inicializar el workspace (tu app, no este repo)
  uv run zenith init --workspace-dir /ruta/a/tu-app --agent claude
  3. Lanzar la misión desde el directorio de tu app
  cd /ruta/a/tu-app
  claude
  3. Y darle al agente esta instrucción:
  ▎ First read .claude/orchestrator_prompt.md and treat it as your primary role,
  then use Zenith to run this mission. <tu tarea>

  El flujo es: Claude Code actúa como orquestador → lee el estado del proyecto →
  lanza workers/testers via ACP → integra resultados → decide si continuar o
  parar. Requiere Python 3.11+, uv, Node 22+, y Claude Code (o Codex como
  alternativa).

● Exactamente, tu comprensión es correcta pero con un matiz clave:

  Lo que Zenith hace mecánicamente:

  - El zenith init inyecta en tu repo un orchestrator_prompt.md (en .claude/ o
  .codex/) + configura un servidor MCP propio (fastmcp).
  - Ese servidor MCP expone herramientas especializadas: start_project,
  submit_plan, advance_project, etc. para el orquestador, y end_node para los
  workers.

  La diferencia con skills/MCPs normales:

  No es solo "lineamientos de comportamiento" — es una arquitectura multi-agente
  activa. El CLI principal (Claude Code/Codex) actúa como orquestador y, a
  través del MCP de Zenith, lanza subagentes independientes (workers y testers)
  via el protocolo ACP. Cada subagente corre en su propio contexto y reporta de
  vuelta.

  Flujo real:
  Claude Code (orquestador)
    └─ lee orchestrator_prompt.md
    └─ usa MCP Zenith → start_project / advance_project
         └─ lanza worker (claude-agent-acp) en proceso separado
         └─ lanza tester (claude-agent-acp) en proceso separado
         └─ integra resultados → decide continuar o parar

  Y sí, mejora el desempeño: los benchmarks muestran que Zenith logra mejor
  ranking que Claude Code solo, y a menor costo que RALPH — porque fuerza al
  orquestador a buscar huecos repetidamente en vez de declarar la tarea completa
  prematuramente.

❯ Y qué ventaja me supondría usar Zenith vs usar varias sesiones en simultáneo  
  de Claude Code o Codex, yo mismo?                                             

● Buena pregunta. La diferencia honesta:

  Lo que haces tú manualmente:
  - Abres varias sesiones, repartes subtareas, revisas outputs, decides qué
  hacer después, coordinas conflictos entre sesiones.

  Lo que Zenith automatiza:
  1. Stopping discipline — el orquestador decide cuándo una tarea está realmente
  terminada, no cuando el agente cree que lo está. Es el problema central que
  el paper identifica.
  2. Gap-finding forzado — cada iteración, el orquestador relee el estado del
  proyecto y busca activamente qué falta, en vez de asumir que lo anterior fue
  suficiente.
  3. Asignación dinámica — decide solo cuántos workers lanzar, si mandar un
  tester, si replanificar, según el estado actual.
  4. Contexto aislado por subagente — cada worker/tester arranca con contexto
  limpio, evitando la degradación de contexto largo.

  Lo que tú haces mejor:
  - Entiendes el dominio real del problema.
  - Puedes redirigir si el orquestador toma una dirección equivocada.
  - Para tareas cortas o bien definidas, tu overhead manual es menor que el
  costo de Zenith.

  Resumen honesto: Zenith tiene ventaja real en tareas largas, ambiguas o que
  requieren múltiples rondas de verificación — donde el riesgo de "el agente
  cree que terminó pero no terminó" es alto. Para tareas concretas y cortas, tú
  coordinando manualmente es más ágil y barato.

  precisión.

  Read 1 file, listed 1 directory (ctrl+o to expand)

● Respuesta directa y honesta:

  Estado actual: solo claude y codex son soportados oficialmente.

  El código es explícito:

  ProviderName = Literal["claude", "codex"]
  ORCHESTRATOR_PROVIDER_NAMES = ("claude", "codex")
  WORKER_PROVIDER_NAMES = ("claude", "codex")

  ---
  Para los modelos (GLM-5.2, Gemini, DeepSeek, etc.):

  Sí es posible, pero indirectamente, vía variables de entorno que el MCP
  reenvía:

  MCP_ENV_FORWARD_ALLOWLIST = (
      "ANTHROPIC_BASE_URL",   # ← apuntar a un proxy OpenAI-compatible
      "GLM_API_KEY",          # ← GLM tiene soporte explícito
      "GLM_BASE_URL",
      "ZAI_API_KEY",          # ← ZAI también
      "ZAI_BASE_URL",
      ...
  )

  GLM y ZAI tienen slots dedicados, lo que sugiere que ya los usan internamente.
  Para Gemini o DeepSeek necesitarías un proxy OpenAI-compatible y ajustar
  ANTHROPIC_BASE_URL.

  ---
  Para otros CLIs (opencode, antigravity, etc.):

  No hay soporte. Zenith escribe configuración en formatos específicos:
  .mcp.json (para Claude Code) o .codex/config.toml (para Codex). Para otro CLI
  necesitarías añadir un nuevo ProviderDefinition en providers.py con su formato
  de config, y que ese CLI soporte MCP stdio — lo cual es factible si el CLI
  habla MCP, pero requiere modificar Zenith.

  Resumen: modelos alternativos — posible con variables de entorno o proxy. CLIs
  alternativos — no sin modificar el código fuente.
